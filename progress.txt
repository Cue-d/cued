## TODO / Future Improvements

- **Recent-first sync**: Currently syncs oldest messages first (ORDER BY m.ROWID ASC).
  Should change to sync most recent messages first for better UX during initial sync.
  Options: (1) Start cursor at maxRowid and work backwards, (2) Query DESC order with reversed cursor logic.

- **Attachments sync**: Currently disabled by default (syncAttachments: false).
  Considerations:
  - Pros: Full message context, searchable files, complete history
  - Cons: Slow initial sync (file I/O + upload), storage costs, privacy concerns
  - Most attachments are ephemeral (memes, screenshots) - low value for CRM
  - HEIC conversion adds complexity, Safari handles natively but other browsers don't
  Recommendation: Skip for MVP. Consider later as opt-in "sync media" toggle, or
  thumbnails-only with on-demand full file fetch. For CRM use case, text content matters more.

- **Reactions sync**: Infrastructure exists (task 2.2b) but not wired to Convex.
  Considerations:
  - Pros: Social context, conversation fidelity, sentiment signal
  - Cons: Extra queries per batch, reactions change frequently, UI complexity
  - Low value for CRM - knowing someone ❤️'d a message doesn't help follow-ups
  - Tapback types already mapped (❤️👍👎😂‼️❓) but not persisted
  Recommendation: Skip for MVP. Reactions are nice-to-have polish, not core CRM value.
  Could revisit for "social listening" features later.

- **Speed up Mem0 memory extraction**: Currently ~600 msgs/hour (24h for 148k messages).
  Options to speed up:
  1. **Increase batch size**: Current BATCH_SIZE=50. Try 100-200 if Mem0 rate limits allow.
  2. **Parallel processing**: Process multiple contacts concurrently within a batch.
  3. **Smarter filtering**: Skip messages that are unlikely to have useful info:
     - System messages, reactions, empty content
     - Messages shorter than N words (not just 10 chars)
     - Duplicate/very similar messages
  4. **Batch Mem0 API**: Use Mem0's batch endpoint (PUT /v1/batch/) if available for adds.
  5. **Reduce cron interval**: 5 min → 1-2 min if rate limits allow (need to test).
  6. **Priority processing**: Process recent messages first, backfill old ones slowly.
  7. **Conversation-level batching**: Send entire conversation to Mem0 at once instead of
     individual messages (fewer API calls, better context).
  Recommendation: Start with (1) increase batch size and (6) recent-first processing.
  Test Mem0 rate limits empirically before going aggressive.

- **Add Convex message ID to Mem0 metadata**: Store original message _id when extracting memories.
  Benefits:
  1. **Traceability**: Know exactly which message a memory came from
  2. **Context retrieval**: Fetch surrounding messages for richer context when recalling
  3. **Deduplication**: Avoid reprocessing same messages if cursor resets
  4. **Source linking**: Show user "where did you learn this?" with link to conversation
  Implementation:
  - Pass message._id in Mem0 metadata: { contact_id, message_id, conversation_id, sent_at }
  - On memory search, optionally fetch source messages for additional context
  - Could enable "expand context" feature in assistant to show full conversation around a fact
