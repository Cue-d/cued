Files: packages/convex/convex/schema.ts, prd.json, progress.txt
Decisions:
  - Added contacts table with userId, displayName (required), company, notes, importance (all optional metadata)
  - Added contactHandles table with userId, contactId (FK), handleType, handle, platform fields
  - Used handleType union: "phone", "email", "slack_id" for type-safe handle classification
  - Used platform union: "imessage", "gmail", "slack" matching integrations table
  - Created searchIndex on contacts.displayName with userId filter for fast contact search
  - Created by_user_handle compound index on [userId, handle] for O(1) contact resolution during sync
  - Created by_contact index for reverse lookups (all handles for a contact)
Why:
  - Separation of contacts and handles enables cross-platform identity resolution (same person, multiple handles)
  - handle field stores normalized values (E.164 phone, lowercase email) for consistent lookups
  - handleType distinguishes phone vs email vs Slack ID without parsing handle string
  - Search index on displayName enables fast fuzzy contact search in UI
  - by_user_handle index is CRITICAL for sync performance: O(1) "does this phone/email exist?" queries
  - by_contact index enables showing all communication channels for a person in UI
Notes:
  - Schema deployed successfully via `npx convex dev --once` (691ms)

  - Created by_conversation compound index on [conversationId, sentAt] for chronological message retrieval

  - by_conversation index is CRITICAL for message thread query performance: load messages for conversation sorted by time
  - search_content full-text index enables fast message search filtered by user and optionally by conversation
  - Filter fields on search index allow scoping searches to user's data and specific conversations
